## 1. Requirements

### 1.1 Overview

Roomsphere is a campus-focused housing and resale platform designed for students in the Five College Consortium. The project addresses two recurring student problems that are often handled through fragmented channels such as group chats, social media, and informal spreadsheets: finding compatible roommates and selling move-out items at the end of a semester. By putting both workflows into one application, the system reduces friction and keeps housing-related interactions inside a safer student-only environment.

The target users are undergraduate and graduate students at UMass Amherst, Amherst College, Smith College, Hampshire College, and Mount Holyoke College. Key stakeholders include students searching for roommates, students posting room vacancies, students buying and selling used items, and the project team maintaining the platform. The primary objectives of the system are to verify users through approved school email domains, support preference-based roommate matching, provide room-vacancy and move-out-sale posting workflows, and enable in-app messaging so students can move from discovery to contact without leaving the platform.

### 1.2 Features

- Verified student authentication using approved Five College email domains
- Preference-based roommate discovery with compatibility scoring
- Room vacancy posting and browsing for students offering available rooms
- Move-out sale marketplace for used student items
- In-app direct messaging for roommate, room-vacancy, and item inquiries
- Search, filtering, and sorting across roommate and listing views

### 1.4 Non-Functional Requirements

1. Only users with approved Five College email domains shall be allowed to create accounts or submit owner/contact emails for protected student-facing flows. This is verifiable through backend serializer validation and should reject 100% of non-approved domains.
2. The application shall persist structured data across service restarts using PostgreSQL storage. This is verifiable by restarting the Docker services and confirming that users, room vacancies, conversations, and move-out items remain available.
3. Messaging updates shall support live inbox and thread refresh behavior while the backend process is running. This is verifiable by opening two clients and confirming that a new message appears in the recipient inbox/thread without a full page reload.
4. The codebase shall separate concerns between presentation, API handlers, service logic, and persistence models. This is verifiable from the repository structure by inspecting the `web/src`, `api/app/view`, `api/app/service`, and `api/app/models` layers.

### 1.5 Challenges & Risks

1. Matching quality risk: compatibility scoring depends on accurate profile data. If users omit habits or enter misleading preferences, the score may overestimate roommate fit. Mitigation: keep the score transparent, show match reasons, and allow direct messaging before commitment.
2. Messaging scalability risk: the current system uses Django Channels with an in-memory channel layer, which is suitable for development but not ideal for large-scale production traffic. Mitigation: move to Redis-backed Channels or another production-ready realtime transport.
3. Testing and release risk: backend tests are written in `pytest` style, but the default Docker workflow does not currently expose `pytest`, and the frontend test tooling is not fully wired in `package.json`. Mitigation: standardize test commands and add CI-ready scripts before deployment.
4. Time and integration risk: the project combines authentication, matching, listings, and messaging in one codebase, so changes in one area can affect others. Mitigation: continue using feature branches, small pull requests, and manual integration checks.

## 2. Design

### 2.1 Architecture Diagram

```mermaid
flowchart LR
    A[React + Vite Frontend] -->|REST API| B[Django REST Views]
    A -->|WebSocket| C[Django Channels Consumers]
    B --> D[Service Layer]
    C --> D
    D --> E[Django Models / ORM]
    E --> F[(PostgreSQL)]
```

The system follows a layered design. The React frontend handles navigation, forms, filtering, and page rendering. Django view modules expose API endpoints for authentication, roommate matching, listings, and messaging. Service modules contain business logic such as compatibility scoring, conversation serialization, and listing creation. Django models encapsulate the persistent data structure, and PostgreSQL stores the underlying data.

The project demonstrates object-oriented and modular design in several ways:

- Encapsulation: user, message, room vacancy, and move-out listing data are encapsulated in dedicated Django model classes.
- Abstraction: service classes such as `RoommateService`, `MessageService`, `MoveoutService`, and `RoomVacancyService` hide business rules from the views.
- Separation of concerns: frontend UI, backend routing, business logic, and persistence are stored in different modules and directories.
- Reusability: shared serializers and service helpers are reused across multiple endpoints and pages.

Major classes and responsibilities:

- `User`: stores identity, budget, campus, preferences, and profile information
- `RoomVacancy`: stores room-vacancy listing data such as rent, housing type, and lease duration
- `MoveoutItem`: stores marketplace listing data such as category, condition, and price
- `Conversation`: represents a unique conversation between two users
- `Message`: stores chat messages, sender, read state, and timestamps
- `RoommateService`: computes compatibility and prepares roommate result payloads
- `MessageService`: manages inbox summaries, thread payloads, and message creation

### 2.4 Tech Stack With Justification

- Frontend: React with Vite
  React supports component-based UI development and reusable page patterns, while Vite provides a fast development server and simple environment-variable handling. This combination is well suited for interactive forms, routing, filtered listing pages, and state-driven messaging views.
- Backend: Django, Django REST Framework, Channels, and Daphne
  Django provides a structured project layout, ORM support, authentication-related utilities, and mature documentation. Django REST Framework simplifies JSON APIs, while Channels and Daphne make it possible to support WebSocket-based messaging updates alongside HTTP endpoints.
- Database: PostgreSQL
  PostgreSQL is a good fit because the project uses relational data with clear entity relationships among users, listings, vacancies, conversations, and messages. It also supports reliable persistence and is easy to run locally through Docker.
- Deployment/Dev Environment: Docker Compose
  Docker Compose makes the full stack reproducible by orchestrating frontend, backend, and database services together. This reduces onboarding friction for teammates and helps keep local environments consistent.

Although the architecture is effective for local testing and small-scale usage, production deployment would require a more robust channel layer and a more formal infrastructure strategy.

## 5. Discussion

### 5.1 Lessons Learned

One of the most important lessons learned during this project was the value of organizing the team around feature ownership while still maintaining a shared integration workflow. The project was broad in scope, covering authentication, roommate matching, room-vacancy workflows, move-out sale functionality, and messaging. As a result, the team benefited from dividing work into manageable feature areas and using GitHub branches and pull requests to coordinate contributions. This approach made it possible for multiple members to work in parallel while reducing integration conflicts and preserving traceability of changes.

The project also highlighted the importance of clear and regular communication. Because frontend pages, backend APIs, and data models were tightly connected, communication had to focus not only on task status but also on interface assumptions, data-field definitions, and expected user flows. In practice, this meant that communication was most effective when it was concrete and implementation-oriented, for example by clarifying which API payloads the frontend expected or which fields were required by a model. A major lesson from this process is that software engineering communication must be both technical and timely; delays in clarifying assumptions can quickly lead to integration issues even when individual components are implemented correctly.

From a software engineering perspective, the team applied several models, tools, and techniques that strengthened the overall development process. The project followed a branch-based collaborative workflow resembling an iterative and incremental development model, where features were developed independently, reviewed, and merged into a shared branch over time. The team used Git and GitHub for version control, Docker Compose for reproducible local deployment, Django and Django REST Framework for structured backend development, Channels and Daphne for realtime communication support, PostgreSQL for relational data storage, React with Vite for frontend development, and automated testing through `pytest` and Vitest. In addition, the team applied modular design techniques such as separating models, services, views, and presentation logic. Learning to combine these tools and patterns in one project was a significant outcome of the course experience.

Another major lesson concerned verification. Early in development, it is often tempting to focus primarily on feature completion, but the later stages of this project demonstrated that build stability, automated testing, and coverage measurement are equally important to the overall quality of a system. The process of adding reproducible test workflows and improving coverage showed that engineering maturity depends not only on whether a feature exists, but also on whether it can be validated, maintained, and extended reliably.

### 5.2 Limitations

Despite the successful implementation of the project's core features, several limitations remain. The first limitation concerns the scale and robustness of the realtime messaging architecture. The current system uses Django Channels with an in-memory channel layer, which is appropriate for development, demonstration, and small-scale testing, but it is not sufficient for a larger production deployment where multiple backend processes and more durable message delivery would be required. This means that the messaging system is functionally correct for the current project scope, yet it would need architectural strengthening before deployment to a broader user base.

A second limitation relates to the quality of roommate matching itself. The matching logic depends heavily on user-provided profile information such as budget range, smoking preference, drinking preference, sleep schedule, and gender preference. If users provide incomplete, inconsistent, or overly simplified preference data, the resulting compatibility scores may not fully represent real interpersonal compatibility. In other words, the system can support structured matching, but it cannot guarantee socially optimal outcomes. This is a common limitation in preference-based recommendation systems and should be acknowledged as both a data-quality and user-experience issue.

Time constraints also influenced the scope and level of refinement of the final system. Because the project combined several nontrivial modules in a single semester, some engineering tradeoffs were necessary. The team prioritized core functional workflows over more advanced enhancements such as stronger analytics, deeper recommendation logic, production-ready realtime infrastructure, and broader usability refinement. In addition, integration work between frontend and backend components created debugging overhead, especially when endpoint behavior, test tooling, and UI expectations evolved at different times.

Although the project now includes strong automated testing and coverage support, there are still limitations in the depth of some frontend interaction tests and branch coverage. The current coverage results indicate that the main logic paths are well tested, but certain alternative UI states, failure conditions, and edge cases could still be exercised more extensively. Finally, the system has not yet been evaluated under significant user load or real-world campus-wide adoption, so performance under peak usage remains an open question rather than a fully validated property.

### 5.3 Future Development Plans

Future development should focus on both technical maturity and user-facing capability. On the technical side, one of the highest-priority enhancements would be replacing the current in-memory messaging layer with a production-ready solution such as Redis-backed Channels. This would improve the reliability and scalability of realtime messaging and would make the platform more suitable for deployment beyond a classroom setting. Additional infrastructure improvements should include continuous integration support for automated test execution, linting, build validation, and coverage reporting so that future changes can be evaluated more systematically.

On the product side, the roommate-matching workflow could be expanded with richer profile information and more nuanced recommendation logic. For example, future versions could support stronger preference weighting, optional personality or lifestyle questionnaires, saved roommate searches, and clearer match explanations. These additions would make the recommendation process more transparent and potentially more useful to students who are making important housing decisions.

The room-vacancy and move-out-sale modules could also be extended with practical marketplace features such as image galleries, listing-status indicators, favorites, reporting tools, and improved filtering and sorting options. Better mobile responsiveness and accessibility support should also be prioritized so that the application remains usable across different devices and for a broader range of users. Because students often browse housing and resale listings from phones, mobile-first refinements would significantly increase the practical value of the platform.

In the longer term, the project could evolve into a more comprehensive student housing ecosystem. Potential future directions include campus-specific moderation rules, notification systems, school-specific verification enhancements, analytics on listing activity, and integration with institutional or student-community channels. Overall, the long-term vision is to move from a strong course project prototype toward a reliable, secure, and scalable student platform that supports discovery, communication, and trust.

### 5.4 Development Process Reflection

The development process reflected an iterative team workflow in which planning, implementation, integration, and refinement occurred in repeated cycles rather than through a single linear sequence. Sprint planning was handled informally through feature decomposition: the team broke the project into major workstreams such as authentication, roommate finder, room-vacancy workflows, move-out sale functionality, and messaging. This made the overall scope more manageable and allowed progress to be tracked in terms of feature completion and integration readiness. A practical advantage of this approach was that the team could continue making progress on independent modules even when another module was temporarily blocked by debugging or interface clarification.

Between the midpoint stage and the final stage, the project changed in both scope and engineering maturity. At the midpoint, the focus was more strongly centered on implementing the core features and establishing the main user flows. By the final stage, the project had become more complete in terms of integrated functionality, and additional effort had been invested in test infrastructure, Docker-based reproducibility, frontend build stability, and coverage reporting. This shift is important because it reflects a transition from feature construction to system hardening, which is a common and necessary phase in software engineering projects.

Risk mitigation was handled through several practical strategies. First, the use of feature branches and pull requests reduced the risk of direct conflicts on the shared branch and created a clearer history of changes. Second, modular separation between models, services, views, and frontend components reduced the impact of local changes and made debugging more targeted. Third, automated testing and coverage improvements helped reduce regression risk by making key behaviors executable and repeatable. Finally, Docker-based environment management reduced setup inconsistency and made it easier to verify that the system worked in a shared runtime context.

Overall, the development process showed that successful team software engineering depends on more than dividing tasks. It requires coordination of architecture, interfaces, testing, and documentation over time. The strongest process lesson from this project is that implementation quality improves substantially when planning, communication, verification, and integration are treated as continuous activities rather than as final-stage tasks.
